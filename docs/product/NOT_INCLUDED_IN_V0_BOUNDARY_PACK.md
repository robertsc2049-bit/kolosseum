# Not Included in v0 Boundary Pack

Status: current v0 communication pack  
Audience: founder, sales, demo, product, engineering  
Purpose: explicit excluded coach expectations for current v0

## Why this exists

Good boundaries close deals faster than vague promises.

This pack makes current v0 non-inclusions explicit so they are easy to communicate and hard to misstate.
It is not hidden engine law.
It is a plain-language boundary surface backed by the machine-readable exclusion registry and CI contradiction guard.

## What v0 is

Current v0 is the deterministic execution alpha for:

- individual_user and coach
- individual and coach_managed execution
- powerlifting, rugby_union, and general_strength
- Phase 1 through Phase 6 only
- factual session execution, split/return, partial completion, coach assignment, factual artefact viewing, and non-binding coach notes

## What v0 is not

The following expectations are **not included in v0**.

| Boundary ID | Expectation a coach may ask for | Included in v0 | Why not included now | Current truthful position |
|---|---|---:|---|---|
| V0-EX-001 | Evidence envelopes, proof export, or sealing | No | Current v0 is Phase 1-6 only; proof layer is outside current ship path | v1/post-v0 |
| V0-EX-002 | Team, unit, gym, or organisation execution runtime | No | Current v0 supports only individual and coach_managed execution | dormant broader platform law / not in current v0 |
| V0-EX-003 | Coach override of engine legality or session decisions | No | Coach authority is observational and non-binding only | forbidden in current model |
| V0-EX-004 | Coach editing athlete Phase-1 truth after declaration | No | Phase-1 declarations are the only lawful truth entry and are immutable for that execution | forbidden in current model |
| V0-EX-005 | Readiness scoring, rankings, dashboards, or analytics judgement | No | Current v0 excludes scoring, rankings, and broader dashboard surfaces | not implemented |
| V0-EX-006 | Messaging, chat, or broad communication workflows | No | Current v0 surface is execution-first and minimal | not implemented |
| V0-EX-007 | Medical, rehab, safety, or suitability framing | No | Claims of this kind are outside permitted system language | forbidden claim class |
| V0-EX-008 | Adaptive coaching, optimisation, or auto-progression decisioning | No | Current v0 is deterministic and non-optimising | forbidden / not implemented |
| V0-EX-009 | Billing or tier state changing engine behaviour | No | Product/payment state must not alter engine truth | forbidden in current model |
| V0-EX-010 | Registry editing UI or runtime registry mutation | No | Registries are read-only artefacts in current scope | forbidden in current model |

## Safe founder wording

Use wording like this:

- "v0 is a deterministic execution alpha for individual and coach-managed sessions."
- "Coaches can assign within system limits, view factual artefacts, and leave non-binding notes."
- "v0 does not include evidence export, org runtime, dashboards, messaging, or coach override of engine decisions."

Do not use wording like this:

- "The coach can override the session if needed."
- "The system explains what is best for the athlete."
- "Org rollout is part of v0."
- "v0 includes audit-proof export."
- "Payment tier unlocks deeper engine behaviour."

## Sales / demo rule

If a prospect asks for something excluded here, the truthful answer is one of:

- "Not included in current v0."
- "That sits in broader platform scope, not the current alpha."
- "That belongs to v1/post-v0 proof or org layers, not the present build."

## Final rule

If a current-v0 claim says or implies one of the excluded expectations exists now, that claim is invalid and the contradiction should fail.
